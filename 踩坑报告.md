## 项目背景

合同对比助手是一款专业的 Word/PDF 文档差异对比工具，专为法律工作场景设计。主要功能包括：

- 自动识别文字增删改
- 表格、页眉页脚、脚注、图片识别
- 生成详细改动列表和 Excel 报告
- 支持 Word 和 PDF 格式

## 关键 Bug 与解决方案

### Bug #1: 程序无法启动 - IndentationError

**错误信息：**
```
IndentationError: unexpected indent
  File "core\doc_comparator.py", line 20
    """文档对比器"""
```

**问题原因：**
- `doc_comparator.py` 第 20 行缺少 `class DocumentComparator:` 类定义
- 文档字符串 `"""文档对比器"""` 缩进错误，前面有多余空格

**错误代码：**
```python
import hashlib
import io
    """文档对比器"""  # ← 错误：缩进且缺少类定义
    
    def __init__(self):
```

**正确代码：**
```python
import hashlib
import io


class DocumentComparator:  # ← 添加类定义
    """文档对比器"""  # ← 正确缩进
    
    def __init__(self):
```

**解决方案：**
1. 在第 20 行添加 `class DocumentComparator:`
2. 确保文档字符串缩进正确（4 个空格）
3. 使用 `cat -A` 检查是否有 tab 混用

**教训：**
- Python 对缩进极其敏感，打包后错误信息会模糊化
- 开发环境能运行但打包后崩溃，多半是语法问题
- 使用 IDE 的语法检查功能可以提前发现

### Bug #2: 页眉页脚提取失败 - AttributeError

**错误信息：**
```
AttributeError: '_Header' object has no attribute 'text'
  File "core\doc_comparator.py", line 105
    header_text = section.header.text.strip()
```

**问题原因：**
- `python-docx` 库中，`section.header` 和 `section.footer` 是对象，不是字符串
- 这些对象没有 `.text` 属性
- 需要遍历其中的 `paragraphs` 来获取文本内容

**错误代码：**
```python
def _extract_headers_footers(self, doc, headers_dict, footers_dict):
    for section in doc.sections:
        if section.header:
            header_text = section.header.text.strip()  # ← 错误
```

**正确代码：**
```python
def _extract_headers_footers(self, doc, headers_dict, footers_dict):
    for section in doc.sections:
        if section.header:
            header_text = " ".join([p.text.strip() for p in section.header.paragraphs]).strip()
```

**解决方案：**
1. 遍历 `section.header.paragraphs` 获取所有段落
2. 提取每个段落的 `.text` 属性
3. 用空格连接并去除首尾空格

**教训：**
- `python-docx` 的 API 文档不够详细，需要实际测试
- 页眉页脚是复合对象，不是简单字符串
- 类似的对象还有 `table.rows`、`document.paragraphs` 等

### Bug #3: 位置显示不直观

**问题描述：**
- 详细改动列表中显示「位置：0」「位置：5」
- 用户需要自己计算是第几段（从 0 开始计数）
- 体验不好，容易混淆

**原始代码：**
```python
self.changes.append({
    'type': 'paragraph_delete',
    'position': i1+idx,  # ← 数字，从 0 开始
    'content': self.paragraphs_x[i1+idx]
})
```

**优化代码：**
```python
self.changes.append({
    'type': 'paragraph_delete',
    'position': f"第{i1+idx+1}段",  # ← 人类可读格式
    'content': self.paragraphs_x[i1+idx]
})
```

**效果对比：**

| 修改前 | 修改后 |
|--------|--------|
| 位置：0 | 第 1 段 |
| 位置：5 | 第 6 段 |
| 位置：100 | 第 101 段 |

**教训：**
- 程序员思维（从 0 开始）≠ 用户思维（从 1 开始）
- 位置信息应该直观，不需要用户计算
- 类似的设计：行号、页码、序号都应该从 1 开始显示

## 代码优化经验

### 优化 #1: 版本号动态读取

**问题：**
- 版本号硬编码在多个文件中（`main.py`、`main_window.py`）
- 每次更新版本需要修改多处
- 容易遗漏导致版本不一致

**优化前：**
```python
# main.py
app.setApplicationVersion("1.2.2")

# main_window.py  
self.setWindowTitle("合同对比助手 v1.2.2")
```

**优化后：**
```python
# 创建 get_version() 函数
def get_version():
    try:
        with open('version.json', 'r', encoding='utf-8') as f:
            data = json.load(f)
            return data.get('version', '1.0')
    except Exception:
        return '1.0'

VERSION = get_version()

# 使用动态版本号
app.setApplicationVersion(VERSION)
self.setWindowTitle(f"合同对比助手 v{VERSION}")
```

**好处：**
- 版本号统一管理在 `version.json`
- 更新版本只需修改一个文件
- 运行时自动读取，无需重新编译

## 打包部署问题

### 问题 #1: macOS 无法打包 Windows exe

**现象：**
- 在 macOS 上运行 `build.py` 可以执行
- 但生成的是 macOS 可执行文件，不是 Windows .exe

**解决方案：**
- PyInstaller 打包是平台相关的
- 必须在目标平台（Windows）上打包
- 代码可以在 macOS 开发，但打包要在 Windows

**正确流程：**
1. macOS 开发、调试、修复代码
2. git 同步代码到 Windows
3. Windows 上运行 `python build.py`
4. 生成 `合同对比助手.exe`

---

### 问题 #2: 打包后缺少依赖

**现象：**
- 打包后的 exe 运行时提示 `ModuleNotFoundError`
- 某些模块在开发环境正常，打包后找不到

**解决方案：**
在 `build.py` 的 `hiddenimports` 中添加缺失模块：

```python
hiddenimports=[
    'docx',
    'docx.opc',
    'docx.opc.oxml',
    'docx.oxml',
    'docx.oxml.ns',
    'docx.text',
    'docx.styles',
    'docx.enum',
    'docx.shared',
    'openpyxl',
    'PyQt6',
    'PyQt6.QtCore',
    'PyQt6.QtGui',
    'PyQt6.QtWidgets',
    # ...
]
```

**教训：**
- PyInstaller 无法自动检测所有动态导入
- 需要在 `hiddenimports` 中显式声明
- 调试版 exe（带控制台）有助于查看错误

---

### 问题 #3: 网络问题导致 git 推送失败

**现象：**
```
fatal: unable to access 'https://github.com/...': 
Error in the HTTP2 framing layer
Empty reply from server
```

**解决方案：**
1. 配置 git 使用 HTTP/1.1：
   ```bash
   git config --global http.version HTTP/1.1
   ```

2. 增加缓冲区大小：
   ```bash
   git config --global http.postBuffer 524288000
   ```

3. 使用 `--force` 强制推送（如果远程有冲突）：
   ```bash
   git push --force -u origin main
   ```

**教训：**
- 网络连接 GitHub 可能不稳定
- HTTP/2 有时会出问题，可以降级到 HTTP/1.1
- 多次重试可能成功

## 最佳实践建议

### 1. 代码规范

✅ **应该做的：**
- 使用 4 个空格缩进，不要混用 tab
- 类定义后必须有文档字符串
- 版本号统一从配置文件读取
- 用户可见的序号从 1 开始

❌ **不应该做的：**
- 硬编码版本号在多处
- 使用程序员思维（从 0 开始）显示位置
- 混用 tab 和空格
- 缺少类定义直接写方法

✅ **推荐的目录结构：**
